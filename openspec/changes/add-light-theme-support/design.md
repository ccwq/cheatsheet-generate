## Context

当前站点的视觉基础集中在 [assets/variables.css](E:/project/self.project/cheatsheet-generate/assets/variables.css) 与 [assets/global.css](E:/project/self.project/cheatsheet-generate/assets/global.css)，现状以暗色变量为默认值，页面直接从 `:root` 读取颜色、背景、卡片与滚动条等设计令牌。导航页模板 [templates/nav.template.html](E:/project/self.project/cheatsheet-generate/templates/nav.template.html) 已经稳定依赖这些变量，但系统还没有主题模式状态层，也没有亮色变量组与切换入口。

这意味着本次变更不是单点调色，而是一次跨越全局样式、页面模板和前端交互的横切式改造。目标不是简单“把背景调白”，而是在保留现有暗色质感的同时，补齐一套高完成度亮色视觉系统，并把主题模式变成可持久化、可扩展、可测试的基础能力。

## Goals / Non-Goals

**Goals:**
- 建立统一的主题模式模型，支持 `dark`、`light`、`auto` 三种模式。
- 将现有暗色变量抽象为主题令牌，并为亮色主题补齐完整变量集。
- 在导航页与 cheatsheet 页面中复用同一套主题切换逻辑与视觉令牌。
- 保证首屏主题判定足够早，减少错误主题闪烁。
- 为后续继续扩展更多主题或局部品牌化皮肤保留结构空间。

**Non-Goals:**
- 不在本次引入第三方主题管理框架。
- 不重做整体信息架构或页面布局。
- 不要求所有历史图片资源都在首版中自动完成亮色重绘，只要求提供可回退的适配策略。
- 不修改 Prism 代码高亮的语法处理逻辑，本次只处理其在页面中的主题适配与容器融合。

## Decisions

### 1. 用 `data-theme` 驱动主题而不是复制整套样式文件
- 决策：在 `html` 或 `body` 根节点设置 `data-theme="dark|light"`，并保留一个独立的主题模式值 `dark|light|auto` 用于持久化；CSS 通过 `[data-theme="light"]` 覆盖变量，而不是分拆成两套平行 CSS 文件。
- 理由：当前样式已经高度变量化，继续沿用 CSS 自定义属性能把改动聚焦到令牌层，避免复制大量组件规则并降低后续维护成本。
- 备选方案：
  - 双样式文件切换：实现直观，但会造成重复规则与加载、覆盖顺序复杂度。
  - 组件级 class 覆盖：适合组件框架，但当前站点偏静态模板，颗粒度过细且改造成本更高。

### 2. 将主题拆为“模式状态”和“生效主题”两个概念
- 决策：内部状态保留 `themeMode = dark | light | auto`，渲染时计算 `resolvedTheme = dark | light`。持久化保存模式值；当模式为 `auto` 时，通过 `matchMedia('(prefers-color-scheme: dark)')` 实时计算。
- 理由：这能同时支持手动指定与系统跟随，不会把“自动模式”错误固化为某一时刻的暗色或亮色结果。
- 备选方案：
  - 只保存生效主题：实现简单，但无法表达“跟随系统”语义。

### 3. 在最早可执行时机注入主题初始化脚本
- 决策：在 HTML 模板的 `<head>` 中引入极小的内联初始化脚本，优先读取 `localStorage` 中的主题模式并设置 `data-theme`，随后页面样式再开始渲染。
- 理由：当前全局背景和文字颜色直接依赖 CSS 变量，如果等到页面底部脚本才切换，首屏会出现明显闪烁，尤其亮暗反差较大时更明显。
- 备选方案：
  - 在 `DOMContentLoaded` 后切换：实现容易，但体验不达标。
  - 服务端预渲染主题：本项目是静态站点，不适用。

### 4. 亮色设计令牌采用独立配色，不直接对暗色变量做线性翻转
- 决策：为亮色主题单独设计背景渐变、文本层级、边框、卡片与强调色，使其保持“精美、优雅、高端”的目标气质；代码块区域可保留相对深色的嵌套容器以保证高亮可读性。
- 理由：暗色转亮色不是简单反相。当前暗色风格依赖高饱和蓝青点缀与深色层叠，如果直接取反会显得刺眼、廉价且层级失真。
- 备选方案：
  - 自动色值换算：速度快，但无法保证审美质量。
  - 完全统一浅底浅代码区：整体更轻，但会削弱代码可读性与高亮对比。

### 5. 主题切换入口先放在导航页控制区，并保留内容页复用接口
- 决策：在现有导航控制面板区域新增主题切换控件；公共脚本中抽出主题初始化、切换、持久化与系统监听能力，供导航页和 cheatsheet 页面复用。
- 理由：导航页是高频入口，放在现有控制区最符合用户预期；公共脚本复用可避免未来在每个页面重复实现。
- 备选方案：
  - 每页单独实现切换：上线快，但逻辑分散。
  - 只在导航页支持切换：会导致进入内容页后体验割裂。

### 6. 先覆盖全局变量与核心组件，再补资源型适配兜底
- 决策：第一阶段优先完成变量、背景、卡片、链接、控件、滚动条、代码容器等主题覆盖；对图片、图标、插画这类资源，优先提供容器底色、描边、滤镜或专用 class 的适配钩子。
- 理由：资源适配往往不规则，先确保系统框架正确，再处理个别资产，可避免阻塞主体实现。

## Risks / Trade-offs

- [亮色主题变量覆盖不完整，导致局部仍使用暗色值] → Mitigation：按变量表建立清单，优先审查 [assets/variables.css](E:/project/self.project/cheatsheet-generate/assets/variables.css) 中所有公开令牌并逐项补齐。
- [初始化脚本与后续交互脚本状态不一致] → Mitigation：将主题模式读取与生效主题计算收敛到单一公共函数，避免一处写 `light/dark`、另一处写 `auto` 的分叉逻辑。
- [Prism Tomorrow 在亮色页面中过重或割裂] → Mitigation：允许代码块保留深色容器，并通过外层卡片与边距控制形成有意的视觉对比，而不是强行整体漂白。
- [现有图片或图标在亮色背景下失去边界] → Mitigation：增加资源容器类、描边阴影或背景承托样式，并在任务中单独安排抽样检查。
- [后续新增页面未接入主题机制] → Mitigation：把主题初始化脚本和根属性约定写入模板与公共文档，要求新页面默认继承该机制。

## Migration Plan

1. 盘点当前全局与页面级 CSS 变量，区分哪些是语义令牌、哪些是组件局部变量。
2. 在公共模板与脚本中建立主题模式读取、解析与 `data-theme` 注入机制。
3. 为亮色主题补齐完整变量集合，并调整全局背景、卡片、文字、链接、表单与滚动条样式。
4. 在导航页控制面板加入主题切换 UI，并验证切换、持久化和首屏恢复行为。
5. 将 cheatsheet 页面模板同步接入主题初始化与亮色变量，校验代码块、标签、卡片等关键区域。
6. 做一轮亮色主题视觉回归，重点检查高对比文字、hover/focus、图标与图片资源表现。
7. 在必要时补充文档说明默认模式、存储键名与扩展约定。
