# cookbook 风格 cheatsheet 规则

## 什么时候采用 cookbook 风格

优先命中以下信号：

1. 用户明确说 `cookbook`、`recipe`、`套路`、`工作流`
2. 原始资料主要是 guides、patterns、use cases、playbooks，而不是 API reference
3. 主题更像“工具箱 / 方法论集合 / command pack / agent harness”
4. 读者真正想要的是“遇到某类任务时该怎么做”

如果主题的核心价值是“字段表、参数表、函数签名、命令全集”，则不要切 cookbook。

## cookbook 风格的组织原则

- 用“任务意图”分组，而不是用“对象类型”分组
- 优先写组合动作，而不是零散命令
- 先给入口和判断规则，再给命令细节
- 每个 recipe 尽量能直接执行或直接迁移到真实项目
- 少写背景故事，多写做法、前置判断、风险点

## 推荐章节骨架

可按主题裁剪，但通常优先包含这些卡片：

1. 入口与定位
2. 仓库 / 工具骨架
3. 起手工作流
4. 高频 recipes
5. 模块协作或角色分工
6. 成本 / 风险 / 决策规则

## recipe 写法

每个 recipe 尽量覆盖：

- 何时使用
- 最小命令链 / 操作链
- 前置条件
- 常见坑或替代路径

## 不推荐的写法

- 把 cookbook 写成安装文档
- 只罗列命令，不说明何时使用
- 把目录树逐项解释成百科
- 为了“像 cookbook”硬塞过多场景，导致正文发散
