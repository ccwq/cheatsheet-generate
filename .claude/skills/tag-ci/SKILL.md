# tag-ci

为本仓库统一维护 cheatsheet 标签体系，并保证 `meta.yml`、`tags/tags-define.yml`、`tags/tags-icon.yml` 三者一致。

## 唯一事实源

- 标签分组定义只认 `tags/tags-define.yml`
- 标签图标映射只认 `tags/tags-icon.yml`
- 条目实际使用只写在 `cheatsheets/<topic>/meta.yml` 的 `tags`
- 不在其他技能、脚本说明、模板说明里重复声明标签规则；其他地方只引用本技能

## 处理范围

- 补齐或修正 `meta.yml` 的 `tags`
- 将旧标签、英文标签、近义标签归一到当前体系
- 判断某条 cheatsheet 是否应属于 `自动化工具`、`CLI 工具`、`脚本工具` 等已有标签
- 必要时新增标签，并同步维护标签定义和图标映射

## 使用策略

1. 先读 `tags/tags-define.yml`，确认当前允许使用的标签集合和所属分组
2. 再读 `tags/tags-icon.yml`，确认该标签是否已有图标映射
3. 再读目标条目的 `meta.yml`、正文摘要、`refmap.md` 或项目主页，判断实际能力与场景
4. 优先复用已有标签；只有现有集合明显无法表达时才新建标签

## 选标签规则

- 优先覆盖“平台 / 领域”“工具类型”“使用阶段”“交互形态”四个维度中的高价值标签
- `工具类型` 是首要维度；像 `CLI 工具`、`自动化工具`、`脚本工具`、`AI 辅助工具` 这类标签优先判断
- 标签应反映“这个项目是什么”和“拿它来做什么”，不要只摘正文里的零散关键词
- 单条 cheatsheet 一般保持精简，优先保留 3 到 6 个高辨识度标签
- 避免把描述词、功能点、实现细节直接当标签；例如版本号、具体命令名、局部 API 名称通常不应入标签

## 创建新标签

仅在以下条件同时满足时创建：

- 现有标签确实无法覆盖该概念
- 该概念具有复用价值，预期不只服务单个条目
- 命名足够稳定，不是临时性说法

创建时必须同步执行：

1. 把新标签加入 `tags/tags-define.yml` 的合适分组
2. 把新标签加入 `tags/tags-icon.yml`
3. 在目标 `meta.yml` 中使用新标签
4. 确认没有更合适的旧标签需要替换或移除

## 归一化规则

- 优先使用当前项目主体系中的中文标签
- 历史英文标签如 `Automation`、`CLI`、`Test`、`Extension`、`Web`，应优先收敛到对应中文标签
- 若一个条目同时存在“旧标签 + 归一化后新标签”，保留新标签，移除旧标签
- 若某条目被误标为某工具类型，但它更像交互入口、平台、库或插件，应移除错误工具类型标签

## 与其他技能协作

- `cheatsheet-maker` 在处理 `meta.yml.tags` 时应调用或遵循 `tag-ci`
- `icon-complete-skill` 只管图标文件，不负责标签策略
- 如果新增标签导致导航页筛选或标签图标展示受影响，最后执行 `npm run build:nav`

## 自检

- `meta.yml` 中的每个标签都能在 `tags/tags-define.yml` 中找到
- 新标签都已写入 `tags/tags-icon.yml`
- 没有在别的技能文档中重复维护标签策略
- 标签数量克制，且能支持导航页筛选
- 构建导航页后，新增或修正的标签能正常显示
