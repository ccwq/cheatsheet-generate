# cookbook 风格 cheatsheet 重构指南

这份参考不是讲“cookbook 是什么”，而是告诉你怎样稳定写出一种更像实战速查卡的正文风格。

目标风格可以概括成一句话：

- 先用一句类比或直觉定位，把陌生主题拉到读者熟悉的脑回路里
- 再按真实动作链拆成几个阶段，让读者知道“现在该做哪一步”
- 每一步给最小可用命令、适用场景和风险提示
- 最后用“小贴士 / 高频场景”收尾，帮助读者把内容带回真实工作区

## 这种风格适合什么主题

优先命中以下信号：

1. 读者关心的是“什么时候用什么”，不是“所有参数分别是什么”
2. 主题天然存在动作顺序，例如开始、查看、恢复、清理、排障、收尾
3. 同一个工具有多个高频情境，靠场景区分比靠参数分类更自然
4. 主题可以用一句贴近日常开发的类比先讲明白

典型适用主题：

- Git / CLI 子命令：`git stash`、`git rebase`、`docker compose`
- 工作流工具：CI、代理编排、hooks、脚手架、发布流
- 页面 / SDK / agent 类工具：重点在接入套路、任务链、收尾动作
- 配置集合或方法论集合：重点在决策路径，不在字段字典

不适合硬套这种风格的主题：

- API 字段表
- 函数签名速查
- 纯语法速记
- 配置项百科

如果主题核心价值是“查字段 / 查参数 / 查签名”，保持普通 cheatsheet 风格更合适。

## 风格目标

这种 cookbook 风格不是“文艺化”，而是把信息组织得更接近真实操作流程。

要让读者读完后立刻知道这三件事：

1. 这个东西像什么，主要解决什么问题
2. 我现在所处的场景，对应哪一组动作
3. 这一步最小要敲什么，哪里最容易踩坑

可以类比前端里的“组件使用文档”和“页面联调手册”的区别：

- 普通文档像 props 表，适合完整查阅
- cookbook 像联调手册，适合边做边看

## 核心写法

### 1. 开头先给一个脑内模型

优先用一两句话把主题解释成读者熟悉的东西。

推荐写法：

- `git stash` 像“临时寄存处”
- Page Agent 像“给现有页面挂一个会思考的前端控制层”
- 某个 orchestrator 像“把多步命令打包成一条流水线”

约束：

- 类比要服务于理解，不要为了有趣而跑偏
- 一句话就够，不要展开成长篇背景
- 类比后最好立刻接一个真实使用场景

推荐句式：

- `X 就像一个……，当你……时，它会……`
- `可以把 X 理解成……，最常见的用途是……`
- `X 不是……，而更像……，适合……`

### 2. 按动作链分段，不按参数堆砌

优先把内容拆成读者实际会经历的阶段，而不是“概念 A / 概念 B / 选项 C”。

推荐阶段：

1. 起手
2. 查看
3. 取回 / 修改
4. 清理 / 收尾
5. 常见场景

这类顺序特别适合：

- 有状态变化的命令
- 有输入、观察、变更、回滚的流程
- 有“开始做什么”和“结束别忘了什么”的工具

如果主题不是严格时间顺序，也可以按任务意图拆：

1. 先接入
2. 再约束
3. 再扩展
4. 最后排障

### 3. 每一段先说“什么时候用”，再给命令

不要一上来就扔命令块。先用一句话告诉读者这一段解决什么场景。

推荐节奏：

1. 一句话说明本段用途
2. 列出最小命令 / 操作项
3. 补一句默认行为、差异点或风险点

例如：

- 当你还没写完代码，但需要临时切分支时，先把当前修改存起来
- 工作切回来后，再决定是“取回并删除”还是“只应用不删除”

### 4. 命令项优先用“标签 + 命令 + 解释”

这类风格的核心不是大段代码，而是高密度、可扫读的动作项。

推荐格式：

- `普通寄存：git stash`
- `带备注寄存：git stash push -m "..."`
- `弹出并删除：git stash pop`

写法要求：

- 先写动作标签，再写命令，再补简短解释
- 标签应该是读者脑中的任务词，不是源码术语
- 一条只讲一个决定点，不要把多个分支塞进同一行

适合放成列表的内容：

- 高频单行命令
- 快捷决策
- 默认行为差异
- 可选参数影响

适合放成代码块的内容：

- 多步操作链
- 带注释的最小流程
- 配置片段
- 需要保留换行结构的目录树 / JSON / YAML

### 5. 用“场景贴士”收尾，而不是用“更多选项”收尾

尾部优先帮助读者把知识迁回真实任务。

推荐写法：

- 突然要修 Bug？ 用哪条最短路径
- 工作区乱了想先清台面？ 用哪条命令
- 想保留记录又怕取回后丢失？ 用 `apply` 而不是 `pop`

这一段的作用像前端里的“常见交互分支”说明，能把零散命令重新绑回实际问题。

## 推荐章节骨架

可按主题裁剪，但优先保持下面的组织顺序：

1. 入口与定位
2. 起手动作 / 最小工作流
3. 查看 / 判断 / 观察
4. 变更 / 恢复 / 扩展
5. 清理 / 收尾 / 排障
6. 核心场景小贴士

如果主题更偏大型工具或仓库，也可以替换成：

1. 快速定位
2. 起手工作流
3. 高频 recipes
4. 模块 / 角色分工
5. 风险 / 成本 / 决策规则

## 语言风格约束

### 要有“带路感”

正文要像一个熟悉该工具的人，在旁边告诉你下一步该做什么。

推荐特点：

- 句子短
- 术语后马上给解释
- 允许少量口语，但不能松散
- 默认站在“正在做事”的上下文里写

### 要像实战笔记，不像百科

优先写：

- 为什么此时用它
- 默认会发生什么
- 哪个分支更适合当前场景
- 哪个点最容易出错

少写：

- 冗长背景
- 历史沿革
- 与当前动作无关的完整参数枚举
- 纯概念定义堆叠

### 要克制修辞

可以有类比，但只能点到为止。

不要：

- 每段都玩比喻
- 用夸张语气替代信息密度
- 把 cookbook 写成宣传文案

## 从示例中抽象出的通用模板

下面这种骨架，最适合改写“单一命令 / 小型工具”的正文：

```md
## 快速定位
---
lang: bash
desc: X 就像一个……。当你……时，用它来……
---

- 核心作用：……
- 典型触发场景：……
- 入口命令：`x`

## 1. 起手动作（Start）
---
lang: bash
desc: 当你……时，先做这一步。
---

- `动作 A：x ...`（默认行为说明）
- `动作 B：x ...`（差异点说明）
- `动作 C：x ...`（补充条件）

## 2. 查看状态（Inspect）
---
lang: bash
desc: 做完第一步后，用这一组命令确认状态。
---

- `查看列表：x list`
- `查看详情：x show`

## 3. 取回 / 修改（Apply）
---
lang: bash
desc: 当你准备继续工作时，用这一组动作恢复或调整。
---

- `方式 A：x apply`
- `方式 B：x pop`
- `指定目标：x pop item@{1}`

## 4. 清理与收尾（Cleanup）
---
lang: bash
desc: 工作结束后，别忘了清理历史或确认结果。
---

- `删除单条：x drop ...`
- `全部清空：x clear`

## 核心场景小贴士
---
lang: bash
desc: 不想背命令时，记住下面这些场景映射就够了。
---

- 场景 A：`命令链 A`
- 场景 B：`命令链 B`
```

## 针对大主题的泛化方式

如果主题不是单个命令，而是一整个工具或仓库，不要机械照搬“寄存 / 查看 / 取回 / 清理”。

要保留的是结构意图，不是标题字面值。

映射方式：

- `寄存` -> `起手 / 建立入口 / 初始化`
- `查看` -> `观察状态 / 判断分支 / 理解结构`
- `取回` -> `执行主任务 / 应用能力 / 继续推进`
- `清理` -> `收尾 / 风控 / 排障 / 释放资源`

例如一个 agent / SDK 主题，常见重组方式是：

1. 它是什么，入口在哪
2. 最小接入怎么跑起来
3. 高频场景怎么套
4. 出问题先看哪里
5. 收尾动作别漏什么

## 正反例

### 推荐

- `git stash 就像一个“临时寄存处”`
- `当你正在写代码却突然需要切去修 Bug 时，它就是最短的中转站`
- `弹出并删除：git stash pop`
- `突然要修 Bug？ 先 stash，再切分支，回来后 pop`

这些写法的共同点：

- 有场景
- 有动作顺序
- 有最小命令
- 有明显决策点

### 不推荐

- `git stash 是 Git 提供的一个命令，用于存储工作区状态`
- `stash 有多种选项，下面分别介绍`
- `参数 A、参数 B、参数 C、参数 D……`

这些写法的问题：

- 像术语解释，不像实战指南
- 读者知道“是什么”，但不知道“现在该怎么做”
- 信息顺序不贴近任务流

## 自检清单

成稿前至少检查以下问题：

1. 开头是否在两句话内给出脑内模型和使用场景
2. 正文是否按动作链或任务链组织，而不是按参数平铺
3. 每个大段是否先写“何时用”，再写命令或操作
4. 高频内容是否优先使用“标签 + 命令 + 解释”的可扫读格式
5. 结尾是否有“核心场景小贴士”或等价的场景映射
6. 是否删掉了大段安装、背景、历史和无关参数表
7. 读者是否能在 10 秒内找到“我现在该看哪一段”

如果以上 7 条里有 5 条以上成立，这篇内容通常已经接近目标风格。
